---

## 2. 为什么要做统一数据层
在 D 阶段之前，本地数据职责分散在多个实现中：

- `AppLocalDataSource`
- `DownloadStore`
- `InstallSessionStore`
- `SharedPreferences`

问题主要有三类：

- 存储职责分散
- 任务类数据缺少统一抽象
- JSON 文件与散落配置的扩展性越来越差

## 5. 当前能力边界

### 已具备
- 统一 facade 访问边界
- 结构化实体与 mapper
- 下载任务 / 安装会话 / 配置策略统一接入
- JSON-backed 结构化持久化

### 仍未完全具备
- schema version
- migration
- 并发写保护
- 事务能力
- Room / SQLite 查询能力
- Repository 完全只依赖统一数据层

---

## 6. 收口结论
D 阶段已经完成“第一轮统一数据层收敛”。

这意味着：

- 本地数据的主访问边界已经建立
- 后续增强可以围绕同一套 facade 与实体继续推进
- 结构化持久化已经有实际落地，而不是停留在设计层

---

## 7. 后续建议
建议优先级如下：

1. 为 `JsonBackedLocalStoreFacade` 增加 schema version
2. 增加 migration 能力
3. 增加并发写保护和更稳的写入策略
4. 评估是否引入 Room / SQLite 做查询增强
