# 06. 工程演进与当前阶段判断

## 1. 这份文档现在怎么读

这个编号位文档原本来自早期的 `A ~ C13` 阶段说明，但项目已经继续向后演进。  
因此这份文档不再只描述 A、C 阶段，而是用它来解释整个工程从早期 UI 骨架走到当前 `R1 / R2 / D / M1 ~ M17` 的脉络。

## 2. 早期阶段做了什么

### A：车机风格 UI

完成了车机视觉基调、导航结构和基础页面样式，为首页、详情页、任务中心这些页面打了底。

### A2：统一按钮与状态标签体系

把页面里的主按钮、状态标签和交互风格抽成统一语义，减少页面各自为战。

### C1 ~ C14：主链路业务骨架

这一阶段把下载、安装、升级、任务中心和状态管理从页面逻辑里抽出来，形成了后面 7 个业务模块的基础形态。

关键产物包括：

- 下载任务记录与恢复
- 安装任务展示
- 升级任务编排
- 统一任务中心视图
- 失败码与重试逻辑
- 统计、筛选、空态、批量操作

## 3. 后续阶段把工程推到了哪里

### R1：真实下载器

把下载能力从模拟流程推进到真实 HTTP 下载、Range、分片、校验、元数据恢复。

### R2：真实安装器

把安装链路推进到 `PackageInstaller.Session`、安装会话落盘、系统确认分发、安装中心 Session 视角。

### D：统一数据层

把本地落盘访问逐步收敛到 `LocalStoreFacade` 和结构化实体上，不再完全依赖散落 JSON 和偏好读写。

### M1 ~ M15：多模块与 feature 化

把工程拆成 `app / common / core / data / business / feature-*` 结构，并完成第一轮资源、binding、依赖方向收口。

### M16 / M17：工程治理

统一文档、依赖版本、资源命名和脚手架基线，降低后续继续演进时的噪音。

## 4. 当前成熟度判断

当前工程的成熟度可以概括为：

**业务主链路已经跑通，真实执行能力已接入关键部分，工程结构也基本稳定。**

已经不是：

- 只有页面壳子
- 只有模拟下载安装
- 只有单 app 包结构

也还不是：

- 全量真实 remote / local / system 数据实现
- 全平台验证完成的交付版

## 5. 当前最值得关注的真实边界

- 下载暂停/取消仍然偏“状态层语义”，不是底层网络任务硬中断
- 安装确认与最终完成链路代码已闭环，但真实设备侧还需要继续联调
- 升级模块仍以编排为主，还没有做更深的自动升级/策略窗口能力
- 本地结构化存储已经建立统一入口，但还缺 schema、migration、并发写保护

## 6. 下一步建议

从当前阶段继续往下走，优先级建议是：

1. 加强下载执行控制：活动任务管理、暂停/取消硬中断、重复启动保护
2. 强化安装平台接入：更多系统回调、OEM 差异、设备联调
3. 推进 Repository 真实化：远端应用数据和系统侧能力进一步替换 fake
4. 强化统一数据层：schema version、migration、写入一致性
5. 给主链路补更多自动化测试和设备侧验证
