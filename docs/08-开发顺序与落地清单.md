# 08. 开发顺序与落地清单

本文不再描述早期“先做真实下载器、先补 `gradlew`”那种旧顺序，而是基于当前工程现状给出新的开发优先级。

## 1. 当前已完成

- A / A2：车机 UI 基调与统一按钮状态语义
- C1 ~ C14：下载、安装、升级、任务中心、状态基座
- R1：真实下载器、下载源治理、环境切换
- R2：真实安装器、安装会话、系统确认分发
- D：统一数据层第一轮收口
- M1 ~ M15：多模块与 feature module 化
- M16 / M17：工程治理、资源命名、文档归并
- P1 基础硬化：目录条件请求缓存、OEM 车况 provider 接缝、聚合层关键测试
- P2 生产硬化：系统边界输入校验+测试、下载基地址外化、XML 全量 @dimen/ 化、策略从 config 读取、开发者工具扩展、Mapper 测试、真机回归清单、后端/OEM 验收标准

## 2. 建议的下一阶段顺序

### P0：真实设备联调

- [ ] 在真实设备上验证下载 -> 安装 -> 升级全链路
- [ ] 记录不同 OEM ROM 的系统确认页与安装回调差异
- [ ] 补充安装失败、升级失败的真机回归记录
- [ ] 固化一份设备侧回归清单

### P1：真实目录 API 接入

- [ ] 把当前目录链路接入真实后端地址
- [ ] 根据后端要求补充认证头、灰度头或签名头
- [ ] 对齐目录字段、错误码和缓存策略
- [ ] 验证 304、回源失败、缓存损坏等降级路径

### P1：OEM 车况信号接入

- [ ] 用真实 OEM provider 替换当前 `StaticVehicleStateSignalProvider`
- [ ] 对接驻车、点火、电源或前后台限制信号
- [ ] 明确信号抖动和延迟时的策略刷新行为
- [ ] 补充 OEM 信号映射与回退策略

### P2：上线前治理与运营能力

- [ ] 灰度发布、黑白名单、强制下架、回滚能力
- [ ] 安装失败与升级失败归因看板
- [ ] 曝光、点击、下载、安装成功率埋点
- [ ] 面向后端/OEM 的联调验收文档

## 3. 每次迭代的检查清单

- [ ] 是否破坏了 7 个业务模块边界
- [ ] 是否绕过了 `StateCenter`
- [ ] 是否把业务逻辑写回了 Fragment / Activity
- [ ] 是否直接操作数据源而没有经过业务层
- [ ] 是否在 `app` 壳层新增了本应属于 `business / data / core` 的逻辑
- [ ] 是否引入了新的资源命名和字符串治理倒退

## 4. 当前推荐分工

- 页面与交互：`feature-* + common`
- 下载与安装真实能力：`core + business`
- 数据层与本地存储：`data`
- 状态与策略：`business/domain/state + business/domain/policy`
- 壳层装配与系统桥接：`app`
- 文档维护：`docs`
