# 08. 开发顺序与落地清单

本文不再描述早期“先做真实下载器、先补 `gradlew`”那种旧顺序，而是基于当前工程现状给出新的开发优先级。

## 1. 当前已完成

- A / A2：车机 UI 基调与统一按钮状态语义
- C1 ~ C14：下载、安装、升级、任务中心、状态基座
- R1：真实下载器、下载源治理、环境切换
- R2：真实安装器、安装会话、系统确认分发
- D：统一数据层第一轮收口
- M1 ~ M15：多模块与 feature module 化
- M16 / M17：工程治理、资源命名、文档归并

## 2. 建议的下一阶段顺序

### P1：下载执行控制继续做实

- [ ] 为下载任务建立更明确的活动 Job 管理
- [ ] 让暂停/取消真正影响底层下载执行，而不只是状态标记
- [ ] 防止重复点击触发重复下载
- [ ] 补下载异常与边界场景测试

### P2：安装平台能力继续增强

- [ ] 继续验证系统确认页与最终安装回调的设备表现
- [ ] 增加更多 `PackageInstaller` 失败分类和恢复策略
- [ ] 处理更多 OEM / 系统差异
- [ ] 完善安装中心对会话中间态的展示

### P3：Repository 逐步真实化

- [ ] 把 fake remote 逐步替换为真实远端接口
- [ ] 强化系统数据读取与安装信息同步
- [ ] 明确各数据源的真相边界和回写顺序

### P4：统一数据层加强

- [ ] 增加 schema version
- [ ] 增加 migration 能力
- [ ] 增加并发写保护
- [ ] 评估是否需要 Room / SQLite 承载更复杂查询

### P5：升级能力深化

- [ ] 丰富升级状态和失败分类
- [ ] 强化自动升级与策略窗口
- [ ] 评估差分升级、静默准备、批量升级策略

### P6：任务中心与自动化验证继续收口

- [ ] 统一任务中心筛选、摘要和批量操作表达
- [ ] 为下载、安装、升级主链路补更多单测
- [ ] 增加设备侧验证记录和回归清单

## 3. 每次迭代的检查清单

- [ ] 是否破坏了 7 个业务模块边界
- [ ] 是否绕过了 `StateCenter`
- [ ] 是否把业务逻辑写回了 Fragment / Activity
- [ ] 是否直接操作数据源而没有经过业务层
- [ ] 是否在 `app` 壳层新增了本应属于 `business / data / core` 的逻辑
- [ ] 是否引入了新的资源命名和字符串治理倒退

## 4. 当前推荐分工

- 页面与交互：`feature-* + common`
- 下载与安装真实能力：`core + business`
- 数据层与本地存储：`data`
- 状态与策略：`business/domain/state + business/domain/policy`
- 壳层装配与系统桥接：`app`
- 文档维护：`docs`
