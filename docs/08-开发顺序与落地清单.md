# 开发顺序与落地清单

本文用于指导后续继续迭代时的推荐顺序，避免多人并行时把边界打乱。

## 一、当前已完成阶段
- A：车机风格 UI
- A2：统一按钮与状态标签体系
- C1 ~ C13：任务中心、统计、筛选、批量操作、安装任务视图、文档体系
- C14：安装中心独立页面（本次新增）

## 二、建议的后续顺序

### P1：真实下载器
- [ ] 用真实 `FileDownloader` 替换模拟实现
- [ ] 补网络失败重试、超时、中断恢复
- [ ] 补更真实的文件校验

### P2：真实安装器
- [ ] 用真实 `PackageInstaller` 替换模拟实现
- [ ] 补安装回调桥接
- [ ] 补安装会话状态恢复

### P3：数据层落地
- [ ] 用 Room/数据库替换当前轻量 JSON 存储
- [ ] 补远端应用列表接口
- [ ] 补首页推荐和搜索真实数据源

### P4：策略层增强
- [ ] 仅 Wi‑Fi 下载开关
- [ ] 夜间自动升级开关
- [ ] 驻车才允许安装/升级的细粒度策略

### P5：任务中心统一基座继续抽象
- [ ] 统一顶部操作组件
- [ ] 统一空态/失败态组件
- [ ] 统一任务卡片基类

## 三、每次迭代的检查清单
- [ ] 是否破坏 7 个模块边界
- [ ] 是否绕过了 `StateCenter`
- [ ] 是否把按钮逻辑散回 Fragment/Adapter
- [ ] 是否引入了 Hilt / Navigation（禁止）
- [ ] 是否保持 FragmentManager + AppContainer 架构

## 四、推荐的团队分工
- UI / 交互：feature + common/ui
- 下载 / 安装：core + domain
- 数据层：data + repository
- 策略与状态：domain/policy + domain/state
- 文档维护：docs/
