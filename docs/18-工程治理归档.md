# 18. 工程治理归档

> 本文档合并原 21/24/26/27 四份工程治理文档，记录多模块拆分、Feature 模块化、构建收口、文档归并和资源命名治理的成果。

## 1. 多模块拆分（M1 ~ M7）

M1 ~ M7 定义了当前工程的多模块结构基础：

| 阶段 | 完成内容 |
|------|---------|
| M1 拆分设计 | 明确 app / common / core / data / business 职责边界 |
| M2 工程骨架 | 拆出基础模块壳子 |
| M3 代码迁移 | 通用能力、数据层、业务编排从 app 迁出 |
| M4 壳层收平 | AppContainer、MainActivity 回归壳层定位 |
| M5 包名收口 | 统一到 com.nio.appstore.* |
| M6 编译校验 | 处理 R、资源归属、包引用问题 |
| M7 文案清理 | Kotlin 文案整理，为后续文案体系打底 |

**价值**: 后续所有真实能力演进（下载器、安装器、数据层）都有了承载位置。

## 4. 资源命名治理（M17）

- 字符串资源命名规范化，feature / common / app 引用同步改名
- 公共 UI 文案向 `common` 收拢
- `app` 不再重复持有公共文案
- 文档归并成稳定入口（R1 文档统一收口到 18 号文档）

## 5. 长期约束

以下纪律需持续保持：

- 新依赖继续走 version catalog（gradle/libs.versions.toml）
- 新文档继续按总览/模块/阶段/交接结构收口
- 新增资源 id 继续保持可读命名
- 公共资源不散回 app 或各 feature 私有副本
- 动态模板不在页面代码中硬编码
- feature 模块依赖方向不能反转
- `app` 模块不能被重新业务化
