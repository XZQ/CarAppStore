# 25. 当前项目状态与接手指南

## 1. 文档目的

这份文档不是阶段归档，而是面向“第一次接手当前工程的人”的单页状态入口。  
目标是让读者在最短时间内回答 4 个问题：

- 这个项目现在做到哪了
- 当前结构是什么
- 最近刚完成什么
- 下一步最值得做什么

## 2. 一页看懂当前状态

这是一个车载应用商店 Android 工程，当前采用：

- MVVM
- `Activity + FragmentManager`
- `AppContainer` 手动依赖注入
- `app / feature-* / business / data / core / common` 分层

固定不变的约束：

- 不使用 Hilt
- 不使用 Navigation
- 7 个业务模块边界不变

一句话判断：

**主链路已跑通，多模块已收敛，真实下载和真实安装第一轮已接入，当前进入“把执行控制、真实数据和设备联调继续做实”的阶段。**

## 6. 最近一次关键进展

最近已提交的关键代码变更是：

- `5f64014`
- `feat: close install user confirmation flow`

这次变更的意义是：

- 系统安装确认页拉起后，安装链路不会提前结束
- 安装器会继续等待最终成功或失败结果
- 安装确认的后半段闭环已经补齐

## 7. 当前最重要的已知事实

### 下载

- 已经不是模拟下载主导，而是真实下载器优先、模拟兜底
- 支持 Range、分片、合并、校验和恢复元数据
- 暂停/取消仍需继续做实

### 安装

- 已经接入系统 `PackageInstaller.Session`
- 已能处理中间态 `PendingUserAction`
- 壳层统一拉起系统确认页
- 确认后会继续等待最终回调

### Repository

- 业务层看到的是稳定 repository 边界
- 当前装配实现仍是 `FakeAppRepository`
- 但内部已是 fake remote + real local/system 的混合状态

## 8. 当前边界与风险

- 下载暂停/取消还没有完全做成底层执行级硬中断
- 下载任务并发控制仍有继续加强空间
- 安装链路真实设备和 OEM 差异还需要继续联调
- 统一数据层已经有 facade，但还缺 schema、migration、并发写保护
- 远端和部分系统能力仍未完全真实化

## 9. 当前最值得继续做的方向

1. 下载执行控制继续做实：暂停/取消硬中断、活动任务管理、重复启动保护
2. 设备侧联调系统安装确认与最终结果回调
3. Repository 逐步真实化
4. 统一数据层继续增强
5. 给下载、安装、升级主链路补更多测试

## 11. 当前状态文档的最佳实践

这类“保存当前进度”的文档，最适合写成一页式状态文档，而不是阶段流水账。  
推荐固定包含这些部分：

1. 文档目的
2. 一句话阶段判断
3. 当前完成度
4. 当前工程结构和技术基线
5. 最近一次关键进展
6. 当前已知边界和风险
7. 下一步优先级
8. 建议阅读顺序

这样做的好处是，新读者先看一页就能知道：

- 当前不是从零开始
- 哪些已经完成
- 哪些只是第一轮
- 继续开发时最该接哪里
