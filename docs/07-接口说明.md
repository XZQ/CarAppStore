# 07. 接口说明

本文只描述当前工程里真正稳定的业务边界，不再沿用“模拟实现即将替换”的旧说法。

## 1. `AppManager`

作用：给页面输出聚合后的视图模型和任务中心数据。

当前核心接口：

- `getHomeApps()`
- `getAppDetail(appId)`
- `getMyApps()`
- `getHomeAppViewData(appId)`
- `searchApps(keyword)`
- `getDownloadManageApps()`
- `getDownloadTasks()`
- `getInstallTasks()`
- `getUpgradeTasks()`
- `getDownloadTaskStats()`
- `getInstallTaskStats()`
- `getUpgradeTaskStats()`
- `getPolicyPrompt()`
- `openApp(packageName)`

边界说明：

- 负责聚合，不负责下载、安装、升级执行
- 依赖 `Repository + StateCenter + InstallSessionStore`
- 页面看到的大部分卡片数据都应该从这里拿，而不是自己拼

## 2. `DownloadManager`

作用：下载业务编排入口。

当前核心接口：

- `startDownload(appId)`
- `pauseDownload(appId)`
- `resumeDownload(appId)`
- `cancelDownload(appId)`
- `removeTask(appId, clearFile)`
- `clearCompletedTasks()`
- `retryFailedTasks()`
- `getPreferences()`
- `updatePreferences(preferences)`

边界说明：

- 负责策略检查、任务记录、状态推进、恢复、重试、偏好控制
- 不负责直接决定页面展示样式
- 底层执行交给 `FileDownloader`

## 3. `InstallManager`

作用：安装业务编排入口。

当前核心接口：

- `install(appId)`
- `clearFailed(appId)`

边界说明：

- 负责 APK 校验、安装状态推进、会话记录同步
- 负责处理 `PendingUserAction` 等系统确认中间态
- 底层执行交给 `PackageInstaller`

## 4. `UpgradeManager`

作用：升级业务编排入口。

当前核心接口：

- `startUpgrade(appId)`
- `checkUpgrade(appId)`
- `checkAllUpgrades()`
- `startBatchUpgrade(appIds)`

边界说明：

- 本质是编排器，不是独立下载器/安装器
- 通过轮询 `StateCenter` 把下载和安装结果折叠成升级状态
- `checkAllUpgrades` 批量检测所有已安装应用的升级
- `startBatchUpgrade` 串行执行批量升级，每步独立策略门控

## 5. `StateCenter`

作用：运行态单一真相。

当前核心接口：

- `observe(appId)`
- `snapshot(appId)`
- `observeAll()`
- `syncInstalled(appId, versionName)`
- `updateDownload(...)`
- `updateInstall(...)`
- `updateUpgrade(...)`
- `resetError(appId)`

边界说明：

- 保存运行态，不等于持久化真相
- 页面、`AppManager`、升级编排都会读这里
- 任何下载/安装/升级状态更新都应该经过这里

## 6. `PolicyCenter`

作用：前置策略判断层。

当前核心接口：

- `canDownload(appId)`
- `canInstall(appId)`
- `canUpgrade(appId)`
- `getSettings()`
- `updateSettings(settings)`

边界说明：

- 当前主要做 Wi‑Fi、存储空间、开关类本地策略
- 结果用 `PolicyResult(allow, reason)` 统一表达
- 不负责真正执行动作

## 7. `FileDownloader`

作用：文件下载底层协议。

当前核心接口：

- `download(request, onEvent)`

当前事件：

- `Waiting`
- `MetaReady`
- `Running`
- `Completed`
- `Failed`

当前实现：

- `RealFileDownloader`
- `SimulatedFileDownloader`

边界说明：

- 只解决文件下载问题
- 不直接关心页面态、业务动作、主按钮文案

## 8. `PackageInstaller`

作用：安装执行底层协议。

当前核心接口：

- `install(request, onEvent)`

当前事件：

- `Waiting`
- `SessionCreated`
- `Progress`
- `PendingUserAction`
- `Installing`
- `Success`
- `Failed`

当前实现：

- `RealPackageInstaller`
- `SimulatedPackageInstaller`

边界说明：

- 只解决安装执行、会话推进和系统确认中间态
- 页面拉起系统确认页不是它的职责，而是通过 `InstallUserActionDispatcher` 交给壳层

## 9. `Repository`

作用：业务层稳定数据入口。

当前事实：

- 业务层依赖的是 `AppRepository` 边界
- 生产装配实现是 `RealAppRepository`，聚合了真实本地存储、系统数据源和 remote 数据源
- `FakeAppRepository` 仅用于测试环境，提供可控延迟的模拟行为
- `AppSystemDataSource` 已接入 PackageManager，支持 openApp、isPackageInstalled、getInstalledVersion、queryInstalledApps
