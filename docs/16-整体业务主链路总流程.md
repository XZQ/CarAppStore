# 整体业务主链路总流程

## 1. 文档目的
这份文档从全局视角描述车载应用商店项目当前已经跑通的主业务链路，帮助团队理解：

- 用户从首页进入商店后的主路径
- 下载、安装、升级三条主链路如何串联
- 应用管理模块、状态中心、策略中心、Repository 分别处在什么位置
- 当前哪些能力已经是真实实现，哪些还处在工程化阶段

全链路里最关键的分工是：

1. `DownloadManager / InstallManager / UpgradeManager` 负责业务执行。
2. `StateCenter` 负责统一页面运行态。
3. `AppManager` 负责把底层结果翻译成页面模型。
4. `Repository` 负责统一数据入口。

## 3. 用户主路径总览

从用户视角看，当前项目最重要的主路径是：

1. 首页浏览应用
2. 进入详情页
3. 点击下载
4. 下载完成后安装
5. 安装完成后打开
6. 在“我的应用”中查看状态变化
7. 在“升级中心”中进行升级
8. 在“下载中心 / 安装中心 / 升级中心”中集中处理任务

## 4. 首页 -> 详情页 -> 下载 -> 安装 主链路

```mermaid
sequenceDiagram
participant U as 用户
participant HOME as 首页/详情页
participant VM as ViewModel
participant DM as 下载模块
participant IM as 安装模块
participant SC as 状态中心
participant AM as 应用管理模块
participant MY as 我的应用/任务中心

U->>HOME: 浏览首页应用
U->>HOME: 点击应用卡片进入详情
U->>HOME: 点击下载
HOME->>VM: onPrimaryClick
VM->>DM: startDownload(appId)

DM->>SC: 更新下载状态/进度/错误
SC-->>HOME: 刷新按钮和进度
AM->>SC: 读取聚合状态
AM-->>MY: 输出任务中心模型

U->>HOME: 下载完成后点击安装
HOME->>VM: onPrimaryClick
VM->>IM: install(appId)
IM->>SC: 更新安装状态

alt 系统要求确认
    IM->>SC: PENDING_USER_ACTION
    SC-->>HOME: 显示等待系统确认
end

alt 安装成功
    IM->>SC: INSTALLED
    SC-->>HOME: 主动作变为 OPEN
    SC-->>MY: 我的应用显示已安装
else 安装失败
    IM->>SC: FAILED
    SC-->>HOME: 主动作变为 RETRY_INSTALL
end
```

## 5. 升级主链路

```mermaid
sequenceDiagram
participant U as 用户
participant UPG as 升级中心/详情页
participant VM as ViewModel
participant UM as 升级模块
participant DM as 下载模块
participant IM as 安装模块
participant SC as 状态中心

U->>UPG: 点击升级
UPG->>VM: onPrimaryClick
VM->>UM: startUpgrade(appId)

UM->>SC: updateUpgrade(UPGRADING)
UM->>DM: startDownload(appId)

loop 等待下载状态
    UM->>SC: snapshot(appId)
end

alt 下载失败
    UM->>SC: updateUpgrade(FAILED)
else 下载成功
    UM->>IM: install(appId)

    loop 等待安装状态
        UM->>SC: snapshot(appId)
    end

    alt 安装成功
        UM->>SC: updateUpgrade(NONE)
    else 安装失败
        UM->>SC: updateUpgrade(FAILED)
    end
end

SC-->>UPG: 刷新升级页
```

当前升级模块本质上是编排层，依赖下载和安装模块完成真实动作。

## 6. 任务中心主链路

当前项目中已经形成 3 类任务中心：

- 下载中心
- 安装中心
- 升级中心

这些页面的共性是：

- ViewModel 不直接拼接底层数据
- 都通过 `AppManager` 取任务视图模型
- 都通过 `StateCenter.observeAll()` 感知变化
- 顶部控制区本质上是在调用对应业务模块能力

### 统一任务中心聚合图

```mermaid
flowchart TD

VM --> AM[AppManager]

AM --> UTV[UpgradeTaskViewData]
AM --> STATS[TaskCenterStats]

STATS --> UI[任务中心UI]
DTV --> UI
ITV --> UI
UTV --> UI
```

## 7. 当前主链路中的公共能力位置

### 7.1 应用管理模块

位置：

- 在 ViewModel 和 Repository / StateCenter 之间

职责：

- 把底层数据翻译成页面模型
- 聚合任务、统计和策略提示

### 7.2 状态中心

位置：

- 在下载、安装、升级三条执行链路中间

职责：

- 承接业务执行结果
- 输出统一 `AppState`
- 推导统一按钮态和状态文案

### 7.3 策略中心

位置：

- 在下载、安装、升级之前

职责：

- 统一做前置拦截
- 返回 allow / reject + reason

### 7.4 Repository

位置：

- 在业务模块和数据源之间

职责：

- 聚合 remote / local / system
- 提供任务记录、偏好、策略、版本、APK 路径等数据

## 10. 当前整体判断

如果按当前代码而不是按早期文档判断：

- 下载：真实能力已接入，但还不是成熟下载平台
- 安装：真实 Session 已接入，但还不是成熟安装平台
- 升级：业务编排已跑通，但还是 orchestration 层
- 状态 / 策略 / 应用管理 / Repository：都已经形成稳定骨架

当前整体阶段更准确的说法是：

**主业务闭环已经跑通，工程已经进入“真实能力接入后的平台化增强阶段”。**
