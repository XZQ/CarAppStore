# M8-M12 任务中心文案收口与最终策略

## 1. 文档目的
本文件合并原先 M8 ~ M12 的任务中心文案治理文档，统一说明：

- 业务层文案第一轮收口
- adapter / formatter 文案继续资源化
- `TaskCenterText` 引入与动态摘要清理
- 残余动态文案收平
- 最终策略定稿

---

## 2. 阶段目标
M8 ~ M12 的核心目标不是“把所有文案都塞进 `string.xml`”，而是为任务中心建立一套稳定、可解释、跨页面复用的文案边界。

---

## 3. 关键结果

### M8：第一轮收口
已完成：

- 新增 `BusinessText`
- 业务层状态 / 策略文案第一轮收口
- 任务中心 fragment 第一轮资源化

### M9：adapter 与 formatter 收口
已完成：

- adapter 文案继续资源化
- `TaskCenterUiFormatter` 继续承担公共组合逻辑

### M10：公共模板引入
已完成：

- 新增 `TaskCenterText`
- 动态摘要与公共模板进一步集中

### M11：残余文案收平
已完成：

- 任务中心 fragment 剩余动态按钮 / 摘要文案继续收口
- `TaskCenterText` 继续收平

### M12：最终策略定稿
最终确定两层策略：

1. 页面专属文案继续放在 `string.xml`
2. 任务中心跨页面公共模板保留在 `TaskCenterText`

---

## 4. 当前职责划分

### `string.xml`
适合承载：

- 页面专属标题
- 局部按钮文案
- 页面级静态说明

### `TaskCenterText`
适合承载：

- 任务中心跨页面公共模板
- 强依赖运行时参数拼装的模板文案

### `TaskCenterUiFormatter`
适合承载：

- 参数组合
- 文案选择
- 对 UI 暴露统一生成入口

---

## 5. 收口结论
`TaskCenterText` 现在不是临时过渡物，而是任务中心公共模板文案层。

这样做更合理，因为：

- 模板文案本身高度依赖运行时参数
- 它们跨多个任务中心共用
- 在多 module 结构下，保留代码型模板比强行全资源化更稳

---

## 6. 后续建议
后续继续做文案治理时，遵循以下原则：

1. 页面专属文案优先走 `string.xml`
2. 任务中心跨页面模板继续走 `TaskCenterText`
3. 不再回退到 fragment / adapter 内部散落拼字符串
