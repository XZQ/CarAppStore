# 22. 任务中心文案体系与治理原则

## 1. 当前结论

这份文档总结的是任务中心文案治理阶段沉淀下来的长期规则。  
这套策略今天依然有效，而且已经不只是任务中心内部策略，而是整个业务文案治理的基础。

## 2. 当前文案边界

### `string.xml`

适合承载：

- 页面专属标题
- 静态按钮文案
- 不依赖复杂运行时参数的提示

### `TaskCenterText`

适合承载：

- 任务中心跨页面公共模板
- 需要运行时参数拼装的摘要和组合文案

### `BusinessText`

适合承载：

- 业务层状态文本
- 下载、安装、升级、策略等通用业务文案

### `TaskCenterUiFormatter`

适合承载：

- 参数组合
- 文案选择
- 统一输出 UI 可消费内容

## 3. 这套策略为什么还成立

当前工程已经 feature module 化，如果把所有动态模板都强行塞回 XML 资源，维护成本会明显上升。  
因此今天最合理的做法依然是：

- 静态页面文案走资源
- 跨页面动态模板保留代码型文案入口
- formatter 负责把业务状态翻译成页面可用文本

## 4. 当前已看到的收益

- 页面层少了很多散落拼字符串
- 任务中心三页文案表达更一致
- 业务层状态文案更容易统一修改

## 5. 当前仍需保持的纪律

- 不要把动态模板回退到 Fragment / Adapter 里拼接
- 不要让业务层直接散落硬编码中文
- 新增跨页面模板优先放到统一文案入口，而不是各 feature 私有复制一套

## 6. 一句话总结

M8 ~ M12 的成果已经成为当前项目的长期文案边界，不是一次性的阶段性清理。
