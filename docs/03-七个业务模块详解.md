# 03. 七个业务模块详解

这 7 个模块构成了当前工程业务层的主骨架：

1. 下载模块
2. 安装模块
3. 升级模块
4. 应用管理模块
5. 状态中心
6. 策略中心
7. Repository

---

## 1. 下载模块

### 职责

- 创建和恢复下载任务
- 校验下载前策略
- 驱动底层下载器
- 持久化任务记录、分片记录、APK 路径
- 更新下载状态、进度和错误信息
- 与安装链路衔接

### 不负责

- 页面模型聚合
- 安装执行
- 升级版本判断

### 当前实现特征

- 业务编排层是 `DefaultDownloadManager`
- 底层执行器是 `RealFileDownloader`
- 已支持真实 HTTP 下载、Range、分片、合并、校验
- 当前暂停/取消仍偏任务级，不是运行时强控制

### 一句话定位

**只管下载流程与下载任务真相。**

---

## 2. 安装模块

### 职责

- 安装前策略判断
- 校验 APK 是否存在且有效
- 驱动系统安装 Session
- 维护安装 Session 记录
- 更新安装状态
- 安装成功后写入已安装结果

### 不负责

- 下载执行
- 升级版本判断
- 页面聚合

### 当前实现特征

- 业务编排层是 `DefaultInstallManager`
- 真实安装器是 `RealPackageInstaller`
- 已接入 `PackageInstaller.Session`
- 已支持系统确认页分发
- 有 `InstallSessionStore` 作为安装中心的会话真相

### 一句话定位

**只管安装流程、安装状态和安装 Session 真相。**

---

## 3. 升级模块

### 职责

- 检查版本
- 判断是否可升级
- 持久化目标升级版本
- 编排下载 + 安装
- 维护升级状态
- 暴露批量升级入口

### 不负责

- 具体下载实现
- 具体安装实现
- 任务视图聚合

### 当前实现特征

- 实现是 `DefaultUpgradeManager`
- 当前升级流程通过轮询 `StateCenter` 等待下载和安装完成
- 本质是 orchestration 层，不是独立升级平台

### 一句话定位

**只管升级编排。**

---

## 4. 应用管理模块

### 职责

- 聚合首页应用卡片
- 聚合搜索结果
- 聚合我的应用
- 聚合下载任务视图
- 聚合安装任务视图
- 聚合升级任务视图
- 生成统计和策略提示
- 暴露统一 `openApp()` 入口

### 不负责

- 下载执行
- 安装执行
- 升级执行
- 原始任务持久化

### 当前实现特征

- 实现是 `DefaultAppManager`
- 主要依赖 `Repository + StateCenter + InstallSessionStore`
- 核心价值是把底层原始数据翻译成页面模型

### 一句话定位

**面向页面提供统一应用视图和任务视图。**

---

## 5. 状态中心

### 职责

- 统一维护下载状态
- 统一维护安装状态
- 统一维护升级状态
- 统一维护错误信息、错误码、进度
- 统一推导按钮态和状态文案
- 对页面提供 `observe()` / `observeAll()`

### 不负责

- 持久化任务记录
- 执行下载 / 安装 / 升级
- 查询远端数据

### 当前实现特征

- 实现是 `DefaultStateCenter`
- `StateReducer` 负责把底层状态翻译为 `statusText` 和 `primaryAction`
- 它是运行态单一真相，不是持久化单一真相

### 一句话定位

**全局统一运行态状态源。**

---

## 6. 策略中心

### 职责

- 判断是否允许下载
- 判断是否允许安装
- 判断是否允许升级
- 管理网络、车况、存储策略
- 持久化策略设置

### 不负责

- 执行具体业务
- 聚合页面模型
- 管理复杂规则引擎

### 当前策略示例

- 非 Wi‑Fi 时禁止下载
- 行车状态时禁止安装
- 低存储模式时禁止下载和安装
- 下载还会额外检查设备可用空间

### 一句话定位

**统一前置决策层。**

---

## 7. Repository

### 职责

- 聚合 remote / local / system 数据
- 提供应用详情、升级信息、已安装信息
- 提供下载任务、分片、APK 路径、偏好、策略设置
- 提供 staged target version

### 不负责

- 页面聚合
- 状态推导
- 业务决策

### 当前实现特征

- 接口是 `AppRepository`
- 实现是 `FakeAppRepository`
- 本地持久化已经比较真实
- 远端和系统能力仍偏 fake / stub

### 一句话定位

**统一数据聚合入口。**

---

## 8. 七个模块边界总结

| 模块 | 核心关键词 | 一句话说明 |
|---|---|---|
| 下载模块 | download orchestration + file pipeline | 只管下载流程与下载任务真相 |
| 安装模块 | install orchestration + session pipeline | 只管安装流程与安装会话真相 |
| 升级模块 | upgrade orchestration | 只管升级编排 |
| 应用管理模块 | app/task aggregation | 面向页面提供统一应用/任务视图 |
| 状态中心 | runtime single source of truth | 全局统一运行态状态源 |
| 策略中心 | policy decision | 决策是否允许执行 |
| Repository | data aggregation | 统一数据出口 |

---

## 9. 当前整体理解

如果只记一句话，可以记成：

**下载 / 安装 / 升级负责执行，状态中心负责运行态，策略中心负责前置判断，Repository 负责数据，应用管理模块负责把这些结果翻译成页面模型。**
