# 21. 多模块拆分与工程收口

## 1. 这份文档怎么读

这份文档归纳的是多模块工程化的起点阶段。  
现在项目已经继续往后演进，因此这份文档更适合被理解成：

**“当前多模块结构为什么会长成今天这样”的历史说明。**

## 2. M1 ~ M7 当时解决了什么

### M1：拆分设计

明确了 `app / common / core / data / business` 的职责边界和迁移顺序。

### M2：工程骨架

把单体 app 工程拆出了基础模块壳子。

### M3：第一批代码迁移

把通用基础能力、数据层和业务编排从 app 中迁出。

### M4：app 壳层收平

让 `AppContainer`、`MainActivity` 开始回到“壳层”定位，而不是继续承载大量业务。

### M5：包名与 namespace 收口

统一到 `com.nio.appstore.*`。

### M6：编译与资源校验

处理拆分早期最容易出现的 `R`、资源归属、包引用问题。

### M7：第一轮 Kotlin 文案清理

为后面的任务中心文案收口打底。

## 3. 它在今天的价值

如果没有 M1 ~ M7，后面的这些事基本都会更难做：

- feature module 化
- 真实下载器和真实安装器接入
- 统一数据层
- 文档归并和工程治理

也就是说，M1 ~ M7 的价值不在于“模块拆出来了”，而在于：

**后续所有真实能力演进都有了承载位置。**

## 4. 当前视角下的阶段结果

从今天往回看，M1 ~ M7 已经达到它该有的目的：

- 基础模块结构稳定
- 依赖方向稳定
- 壳层定位清晰
- 为后续 `feature-*` 模块提供了落点

## 5. 当前仍需注意的遗留点

- `app` 仍然是总装配入口，不能被重新业务化
- 共享资源和共享布局仍然要持续保持归属清晰
- 继续新增 feature 时，依赖方向不能反转

## 6. 一句话总结

M1 ~ M7 不是当前开发重点，但它定义了今天整个工程为什么能以多模块方式继续演进。
